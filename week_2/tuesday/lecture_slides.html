Local vs Remote: The Critical Distinction

LOCAL SYSTEM

On Your Computer:

  • Your Files: HTML, CSS, JS on your hard drive
  • Shell/Terminal: Bash, Zsh, PowerShell, etc.
  • Git Software: Version control running locally
  • VS Code: Visual interface to files + Git
Terminal - Bash
user@computer:~$ git status
On branch main
Changes not staged...

REMOTE SYSTEM

In The Cloud:

  • GitHub Servers: Microsoft data centers
  • Remote Repository: Copy of your project
  • Accessed via Internet: Not on your device
  • Backup & Collaboration: Safe, shareable storage

Your repo URL:

github.com/username/project

This lives on GitHub's servers, NOT your computer

CRITICAL: Your computer and GitHub are SEPARATE systems connected by the internet. Git pushes code FROM your computer TO GitHub servers.

Understanding the Shell

The Command-Line Interface to Your OS

Shell: A program that interprets and executes commands. It's the text-based way to interact with your operating system.

Common Shells:

  • Bash (Bourne Again Shell) - Most common on Linux/Mac
  • Zsh (Z Shell) - Default on modern macOS
  • PowerShell - Windows shell (more powerful than cmd)
  • cmd - Old Windows command prompt

Shell Programming (Shell Scripting):

Writing sequences of commands in a text file to automate tasks.

example-script.sh
# This is a shell script
git add .
git commit -m "Auto commit"
git push
echo "Backup complete!"

Why This Matters:

Git is accessed through shell commands. VS Code has a built-in terminal where you can run these commands, OR you can use VS Code's visual Git interface which runs the same commands behind the scenes.

The Complete Git Workflow

Four Distinct Stages

1. Working Directory

Your files on your hard drive. Changes not yet tracked.

2. Staging Area

git add
Choose what to commit

3. Local Repository

git commit
Saved snapshot on your PC

4. Remote Repository

git push
Backup on GitHub servers

The Complete Flow:

Terminal
# 1. Check what changed
$ git status
modified: index.html

# 2. Stage the changes
$ git add index.html

# 3. Commit to local repository
$ git commit -m "Added navigation menu"

# 4. Push to GitHub (remote)
$ git push

VS Code Visual Git Indicators

Understanding the Letter Codes

Source Control Panel:

Changes:

M index.html Modified
U about.html Untracked (new file)
A contact.html Added to staging
D old-file.html Deleted

What Each Indicator Means:

  • M (Modified): File exists in repo but you changed it
  • U (Untracked): New file Git doesn't know about yet
  • A (Added): File staged and ready to commit
  • D (Deleted): File removed from project

Pro Tip: The Source Control icon in VS Code's sidebar shows a number badge - that's how many files have changes!

The .gitignore File

What NOT to Track

Some files shouldn't be in your repository:

  • Operating system files (.DS_Store on Mac, Thumbs.db on Windows)
  • Editor configuration (.vscode, .idea)
  • Dependencies (node_modules folder)
  • Sensitive data (passwords, API keys)

Create a .gitignore file:

# OS Files
.DS_Store
Thumbs.db

# Editor
.vscode/
.idea/

# Dependencies
node_modules/

# Environment variables
.env

Important: Create .gitignore BEFORE your first commit to avoid tracking unwanted files!

Common Git Mistakes (And How to Avoid Them)

Mistake 1: Forgetting to Stage

$ git commit -m "Update"
nothing to commit
Fix: git add . first!

Mistake 2: Committing Without Pushing

Your code is saved locally but NOT backed up to GitHub!

Always run git push after committing to send changes to GitHub

Mistake 3: Wrong Commit Messages

Bad:
"stuff"
"fixed bug"
"updates"
Good:
"Added responsive navigation menu"
"Fixed form validation error"
"Updated hero section styling"

Mistake 4: Editing Files on GitHub Web Interface

This creates conflicts with your local files. Always work locally, then push.

CSS: Three Ways to Style

Understanding Your Options

Remember from Week 1: CSS was created to separate content from presentation. But there are three ways to apply styles:

1. Inline

Style attribute on element

2. Embedded

<style> tag in <head>

3. External

Separate .css file linked

The Question: Which method should you use?

Spoiler: Almost always external! But let's understand why...

12 / 20

Method 1: Inline Styles

Applying Styles Directly to Elements:

<h1 style="color: red; font-size: 32px;">Hello World</h1>
<p style="color: blue; font-size: 16px;">Paragraph</p>
<p style="color: blue; font-size: 16px;">Another paragraph</p>
Pros Cons
Quick for testing single element Repetitive - must style every element
Highest specificity (overrides other styles) Hard to maintain
No external files needed Mixing content with presentation (defeats CSS purpose!)
Can't reuse styles

Use Case: Quick testing only. Avoid in production code!

13 / 20

Method 2: Embedded Styles

Using a <style> Tag in the <head>:

<!DOCTYPE html>
<html>
<head>
  <style>
    h1 {
      color: red;
      font-size: 32px;
    }
    p {
      color: blue;
    }
  </style>
</head>
<body>
  <h1>Hello World</h1>
  <p>All paragraphs get these styles!</p>
</body>
</html>
Pros Cons
Styles all matching elements at once Only affects this one HTML file
Good for single-page projects Can't share styles across multiple pages
No extra files to manage Makes HTML file longer and harder to read

Use Case: Small single-page projects or quick prototypes

14 / 20

Method 3: External Stylesheet (Best Practice)

Separate CSS File Linked to HTML:

index.html:

<!DOCTYPE html>
<html>
<head>
  <link rel="stylesheet"
        href="styles.css">
</head>
<body>
  <h1>Hello</h1>
</body>
</html>

styles.css:

h1 {
  color: red;
  font-size: 32px;
}

p {
  color: blue;
}
Pros Cons
Reusable across ALL pages Extra file to manage
Clean separation of concerns Requires correct file path
Browser caches CSS file (faster loading)
Easy to maintain
Industry standard

Use This! External stylesheets are the professional, maintainable way to style websites.

Linking External CSS: File Paths Matter!

The <link> Element:

<link rel="stylesheet" href="styles.css">

Critical Attributes:

  • rel="stylesheet" - Tells browser this is CSS
  • href="..." - Path to your CSS file

File Path Examples:

<!-- Same folder as HTML -->
<link rel="stylesheet" href="styles.css">

<!-- CSS in subfolder -->
<link rel="stylesheet" href="css/styles.css">

<!-- CSS in parent folder -->
<link rel="stylesheet" href="../styles.css">

Common Mistakes:

Wrong: <link href="styles.css"> - Missing rel attribute!

Wrong: <link rel="stylesheet" href="style.css"> - Typo in filename!

Wrong: <link rel="stylesheet" href="Styles.css"> - Wrong capitalization!

Pro Tip: If your styles don't load, open DevTools (F12) ‚Üí Console tab to see 404 errors!

CSS Selectors: Targeting Elements

Four Main Types:

1. Element Selector:

/* Targets ALL p elements */
p {
  color: blue;
}

2. Class Selector:

/* Targets elements with class="highlight" */
.highlight {
  background-color: yellow;
}
<p class="highlight">This paragraph gets highlighted</p>

3. ID Selector:

/* Targets element with id="header" (only ONE per page!) */
#header {
  font-size: 48px;
}
<h1 id="header">Main Header</h1>

4. Descendant Selector:

/* Targets p elements INSIDE nav elements */
nav p {
  font-weight: bold;
}

Troubleshooting CSS & Git

CSS Not Loading? Check These:

  1. File path: Is href correct? Case-sensitive!
  2. File saved: Did you save styles.css?
  3. Link location: Is <link> in <head>?
  4. Typos: Is rel="stylesheet" spelled correctly?
  5. Browser cache: Hard refresh (Ctrl+Shift+R)
Browser DevTools Console
Failed to load resource: net::ERR_FILE_NOT_FOUND
styles.css:1
This means the file path is wrong!

Git Changes Not Showing on GitHub?

  1. Did you stage? Run git add .
  2. Did you commit? Run git commit -m "message"
  3. Did you push? Run git push
  4. Check branch: Are you on the right branch?

Golden Rule: When in doubt, run git status to see what's happening!
